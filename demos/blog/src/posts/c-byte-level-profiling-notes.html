Hot-path tuning in C is mostly about data layout, not syntax tricks. Once your program does enough work per second, cache miss penalties dominate. A clean codebase can still be slow if the memory access pattern is chaotic. So I profile first, then optimize at the byte-layout level.

I treat structs as I/O contracts with the cache hierarchy. Fields used together belong together. Rarely used flags move to side tables. Boolean-heavy structs can often be compacted into bitfields or packed masks if you benchmark carefully and preserve readability with helpers.

A practical profiling cadence

  1. Capture a baseline on representative input.
  2. Inspect top callers and top cache-miss regions.
  3. Change one layout or branch pattern at a time.
  4. Re-run with the same harness and compare deltas.

Most wins come from deleting accidental work: unnecessary copies, repeated parsing, and branch-heavy conditionals in tight loops. Fancy intrinsics are sometimes useful, but not before the baseline architecture is honest.

Code generation can remove repetition, but it can also hide intent. The worst generator outputs are unstable and difficult to diff; every rebuild rewrites lines, and every review becomes noise. I only keep generators that produce deterministic output with stable ordering and predictable formatting.

A second requirement is traceability. When a generated branch misbehaves, engineers need to map output back to source templates quickly. That means embedding source hints, preserving section boundaries, and documenting generation passes as if they were a compiler pipeline.

Before writing a generator, decide how you will debug failures in production. If that path is unclear, the generator design is incomplete.

What made our generators maintainable

  • Stable sort order for all emitted symbols.
  • One generator pass per concern, rather than monolithic transforms.
  • Fixture tests on generated files, not just source templates.
  • Human-readable error output with source file references.

With those constraints, metaprogramming behaves more like a build tool and less like wizardry. That is the right tradeoff for teams who care about velocity over novelty.

Most internal DSLs fail for one reason: they try to replace the host language instead of complementing it. The result is brittle syntax, awkward error messages, and a steep onboarding curve. I design DSLs as compression layers over repetitive structure, not as alternate universes.

In practice, this means choosing a tiny core with explicit escape hatches. If users need to step outside the DSL for edge cases, they should be able to do so naturally. A DSL without escape hatches becomes a lock-in trap for its own authors.

Three constraints that kept our DSLs healthy

  • Syntax mirrors the host language where possible.
  • Error messages reference source terms the author actually wrote.
  • Generated output remains first-class and testable on its own.

Common Lisp is excellent here because macros let you prototype syntax quickly, but the same discipline applies in Rust proc-macros or C preprocessors: keep the surface small and the emitted artifacts boring.

When done right, a DSL disappears into the workflow. Engineers think about domain intent, not parser quirks.
